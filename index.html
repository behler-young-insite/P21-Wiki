Suppress Pop-Ups

  • Problem: Cannot do certain things via non-interactive api due to pop-ups
  • Solution: Suppress the pop-up by creating a business rule to default it
  • Solution:
    1. RMB --> DynaChange rule --> listen for new rule --> trigger pop-up
    2. DynaChange window opens for rule configuration
    3. Configured rule applies to ALL popup events (select * from p21_view_messages m(nolock)) to see all the popup messages
    4. in order to target your business logic, a series of if/thens are required
      1. if message_no == _____ (specific popup number) - cannot rely on user_text, not a 1:1 match due to dynamic content
      2. set default_button to what I want it to select (e.g. override price anyway? Yes)
      3. set suppress_message to Y so that default is used
      4. consider restricting this rule to api user unless you want to suppress the popup for everyone
        5. 
	public override RuleResult Execute()
	{
		RuleResult Result = new RuleResult();
		try
		{
			DataTable msgbox_info;
			msgbox_info = Data.Set.Tables["MessageBoxData"];
			if (msgbox_info.Rows[0]["message_no"].ToString() == 8333.ToString())
			{
				msgbox_info.Rows[0]["default_button"] = "1";
				msgbox_info.Rows[0]["suppress_message"] = "Y"; // Can set this to Y to suppress message box. Default button
				automatically used.
			}
			Result.Success = true;
		}
		
		catch (Exception ex)
		{
			Result.Message = ex.Message;
			Result.Success = false;
		}
		
		return Result;
	}

Slow API Calls

  • Problem: Api calls are Intermittently slow (slow starts) due to P21 Idling.
  • Solution: A look at changing the IIS setting as outlined in this link.
  • Solution: A bit hacky, but you could ping the api at an interval to keep it from falling asleep

  • Problem: Transaction and Interactive Api calls generally slow.
  • Solution: Could be Dynachange rules slowing you down - make a user for your api calls and exclude them from all non-essential Dynachange rules that you can.
  • Solution: On the website, you could rely on async calls so that the slowness of P21 API is mitigated
  • Solution: If real time is essential and you are on prem with your site and db hosted on same server, you could copy/utilize the P21 stored procedures to get the data you need.

Order Creation post returns 200 Success but does not create an order

  • Problem: Order creation post succeeds but doesn't create an order; an order post should either Fail with a generic message, Fail with a specific message, or Succeed and return the order number of the order Created
  • Potential Cause: The order submitted exceeds a customers credit limit
  • Solution: Raise the customer's credit limit
  • Explanation: while this should fail and send back a message as to why it failed, that is not the case. This is a bug which may/may not be fixed and for the meantime will need to be worked around.

  • Potential Cause: All items on the order do not exist in the sale location
  • Solution: try adding an item that does exist at the selected sales location
  • Explanation: if all the items exist, then the order is created fine. If none of the items exist - then the order ends in suspended records. If some of the items exists, and some do not then the order will be imported and the items the don't exist will just be missing from the order there should be a record indicating that items fell off which you will specifically need to look at. To find it there is a sql table where these can be looked at via `select * from p21_view_pending_import with(nolock)` or in the 'Edit Pending Import Records' window in P21
  • Additionally: Ensure 'Save import Suspense Files to Database' is set to true in P21 at: System Settings -> Imports/Exports -> General
  • Potential Cause: There might be a Dynachange rule or system popup/alert that is preventing it from working
  • Solution: Audit any dynachages / system alerts tied to order submit, disable one by one - perhaps run in debugger with breakpoints and see what variables are in play

How to Create Multiple order lines of identical products

  • Problem: Need to create an order with multiple order lines with variable qty
  • Solution: give products alt part numbers and submit the identical item lines under alternate part numbers
  • Explanation: The P21 API (transaction & interactive endpoints) will accept alternate part numbers as item id's and do the conversion on the fly when creating the order. You can send orders to the API using the alternate part number in place of the item id the API doesn't mind it will automatically convert the item to the part number but oddly enough in this instance it 'skips' the 'merging' so if you sent something like alt1 qty: 20, alt1 qty = 10 in one api call the api would turn that into an order with two lines for the same item id with different quantities
  • Warning: This most definitely seems like a hack, so if Epicor ever fixes it, the solution may no longer be viable

Skinny Receipt Configs

  • Problem: Need to configure skinny receipt printing, but where?
  • Solution: Stolen from this forum thread
    You need to have a valid "skinny" receipt printer setup in the config. Before logging into P21 do Alt+T you will see 2 new fields there for the skinny and wide receipt printers. If those fields are blank it cannot print. Also you can modify the receipt in Dynachange Screen Designer to add columns. It is not crystal. In reprint receipts window do preview then RMB-Dynachange.
